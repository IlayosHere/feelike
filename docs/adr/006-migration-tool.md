# ADR 006 — Migration Tool: Alembic

**Status:** Accepted
**Date:** 2026-04-21

## Context

feelike's backend uses SQLAlchemy 2.x and needs to support two DB targets: SQLite in local dev (instant, zero-setup) and PostgreSQL 16 in prod (Cloud SQL). Raw-SQL migrations diverge between these engines — UUID types, `TIMESTAMPTZ`, `NOW()`, `ON DELETE CASCADE` syntax all differ — making "handwritten SQL" a false economy the moment the schema grows.

We left the tool "TBD" in Phase 0. Resolving it now, before the first migration is written.

## Decision

**Alembic.** Use SQLAlchemy's official migration tool.

- Migration files live in `backend/migrations/versions/`
- Config in `backend/alembic.ini`
- Models are the source of truth; Alembic autogenerates migrations from model diffs (`alembic revision --autogenerate -m "<desc>"`)
- Every autogenerated migration is **human-reviewed** before commit (autogenerate misses enum changes, CHECK constraints, and index renames)
- Migrations are **forward-only** (no `downgrade()` bodies beyond what autogenerate writes; never rely on them in prod)
- Startup runs `alembic upgrade head` automatically in dev; in prod, migrations are a deliberate step (CI job or manual gcloud command) — never auto-run on Cloud Run start

## Rationale

- **SQLite ↔ Postgres parity.** Alembic's SQLAlchemy-aware operations (`op.create_table`, `op.add_column` with `sa.Column(... UUID)`) resolve correctly for both engines. Handwritten SQL requires per-engine branches.
- **Autogeneration speeds iteration.** Change a model, run `alembic revision --autogenerate`, review diff, commit. One command per schema change vs hand-writing DDL twice.
- **Industry standard.** Every FastAPI + SQLAlchemy tutorial uses it. New agents / future collaborators know it.
- **Forex-dashboard uses handwritten SQL and a startup migration shim** — that pattern works for their simpler schema but won't scale for feelike once we add attachments, full-text search, etc. We intentionally diverge here.

## Alternatives rejected

- **Handwritten SQL files in `migrations/NNN_desc.sql`** — the original Phase 0 plan. Fails on SQLite/Postgres divergence; no autogeneration; no dependency graph.
- **Django-style migrations** — we're not using Django ORM.
- **Liquibase / Flyway** — heavier, JVM-based; overkill.
- **No migrations, just `create_all`** — fine for first 30 seconds, then every schema change is destructive.

## Consequences

**Committing to:**
- Add `alembic` to `backend/requirements.txt` in Phase 3 scaffold.
- `backend/alembic.ini` + `backend/migrations/env.py` set up to read `DATABASE_URL` from env.
- Every model change → autogenerated migration + review → commit together.
- `alembic upgrade head` is part of local dev startup (via a `make` target or startup hook).
- In prod, migrations run as a Cloud Run Job or one-off `gcloud run jobs execute` step before the service deploy — NOT in the Cloud Run service's startup, to avoid race conditions with horizontal scaling.

**Not doing:**
- No downgrades used in prod (forward-only policy from coding-standards.md stands).
- No autogenerate-and-commit blindly. Human review is mandatory.

## Update `/add-migration` slash command

The existing `.claude/commands/add-migration.md` assumed handwritten `.sql` files. It now reflects the Alembic flow: run `alembic revision --autogenerate`, review, edit the generated file, commit alongside the model change.

## Revisit when

- We outgrow Alembic (unlikely for this scale).
- We add a second microservice with its own DB — then each service owns its own `alembic/`.
