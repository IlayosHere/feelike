Create a new migration for the feelike backend using Alembic. Args: `$ARGUMENTS` = short description in snake_case (e.g. `add_mood_to_entries`, `create_tags_table`).

See `docs/adr/006-migration-tool.md` for the full rationale.

## Steps

1. **Verify `backend/alembic.ini` exists.** If not, Alembic isn't set up yet — stop and tell the user we need to run the Phase 3 backend scaffold first.
2. **Edit the SQLAlchemy model(s)** in `backend/api/models.py` (or `backend/api/models/*.py` if already split) to reflect the desired schema.
3. **Generate the migration** from repo root:
   ```
   cd backend && alembic revision --autogenerate -m "<$ARGUMENTS>"
   ```
   This creates `backend/migrations/versions/<hash>_<$ARGUMENTS>.py`.
4. **Review the generated file** — **always**. Autogenerate misses:
   - Enum changes
   - CHECK constraints
   - `server_default` adjustments
   - Index renames (it drops + recreates)
   - Data migrations (you have to write them by hand in `op.execute(...)`)
5. **Apply locally to confirm:**
   ```
   cd backend && alembic upgrade head
   ```
   Run the test suite to catch schema/ORM mismatches: `pytest backend/tests`.
6. **Update `docs/data-model.md`** with any schema-shape changes.
7. **Report** — migration file path, summary of the change, any manual edits you made to the autogenerated body, whether the test suite passed.

## Rules

- **Forward-only in prod.** Alembic generates a `downgrade()` function; leave it in for local dev convenience but **never rely on it in prod**. If a prod migration is wrong, roll forward with a new migration.
- **Destructive migrations require a two-step deprecation.** If you're dropping a column or table, stop and confirm with the user first. Then: release N stops writing to it, release N+1 drops it.
- **Every new table gets:** `id` (UUID), `created_at`, `updated_at`. User-owned tables also get `user_id` FK **`ON DELETE CASCADE`** + an index on `user_id`. Entry-owned join tables (e.g. `entry_tags`) also use `ON DELETE CASCADE` on both FKs.
- **Idempotent DDL where Alembic allows** — e.g. `op.create_table` is inherently idempotent only if the table doesn't exist; don't bypass Alembic's version tracking.
- **In prod, migrations run as a Cloud Run Job or one-off `gcloud run jobs execute`** step BEFORE the service deploy. Never auto-run on service startup (race condition under horizontal scale).
